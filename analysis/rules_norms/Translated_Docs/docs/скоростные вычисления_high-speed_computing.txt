SUPERSCALAR MICROPROCESSORS

In superscalar microprocessors, the degree of redundancy of computational nodes is high.
There are several integer and real ALUs, there is a branch prediction block, there is a shadow register file, and a lot of other things are duplicated.
This makes it possible to break sequential code into pieces and execute it in parallel.

MEMORY BARRIERS

A memory barrier is a way of controlling unscheduled execution of instructions by the processor and/or compiler.
It is unintuitive to execute code out of sequence. Code should, after all, be executed in the order in which it is written (it really isn't).
Operations on data not explicitly (!) related to each other may occur either out of order or at all simultaneously
within the same core, utilizing the massive redundancy of the processor's computational nodes.
The memory barrier (to a rough approximation) forces the code to execute as it is written on the screen (sequentially and predictably),
and not as the processor and compiler want (in a counter-intuitive but more optimal order).
That is, it guarantees that the code before the memory barrier is partially or completely executed by the time the barrier instruction completes.
A barrier is usually an assembly instruction (i.e., it is present in the processor's instruction system).
The barrier is usually a "there is an implicit relationship between this data" hint, but not necessarily.

In C++, there are three memory models for atomics:
1. relaxed: only operations are guaranteed to be executed atomically. In what order is the question.
- modification of a variable will "appear" in another thread not at once
- thread2 thread will "see" the values of the same variable in the same order as its modifications in thread1 thread.
- the order of modifications of different variables in thread1 thread will not be saved in thread2 thread.
relaxed variables can be used as counters or stop flags.
The fastest and most unreliable memory model.
The analog from the DBMS transactional model is READ UNCOMMITTED
2. sequential consistency, seq_cst: memory state is synchronized between all program threads.
- the order of modifications of different atomic variables in thread1 thread will be preserved in thread2 thread
- all threads will see the same modification order of all atomic variables.
  The modifications themselves may occur in different threads
- all memory modifications (not only modifications over atomics) in thread1 thread, which executes store on an atomic variable,
  will be visible after performing load on the same variable in thread2 thread
The slowest and most reliable memory model.
The analog from the DBMS transactional model is SERIALIZED
3. acquire/release: pair synchronization.
- a modification of an atomic variable with release will be instantly visible in another thread performing a read of the same atomic variable with acquire
- all memory modifications in thread1, which is writing an atomic variable with release,
  will be visible after reading the same variable with acquire in thread2 thread
- the processor and compiler cannot move memory write operations below the release operation in thread1,
  and cannot move a read operation from memory above an acquire operation in thread2 thread.
Allows synchronization only between two threads (as opposed to all threads in thread1 and thread2).

NANOSECOND TIME MEASUREMENT

Accurate time measurement on x86/x64 architecture is a large topic.
On fixed-frequency scalar processors you can count clock cycles, but on x64 you cannot: the result will be different.

One and the same code may be executed for different times because of different states of system nodes, which is (roughly) determined by the total load on the system.
Determinism is absent because of:
- extraordinary execution of instructions
- superscalar architecture (pipeline, shadow registers, cache state and other duplication)
- floating frequency of both the processor and separate parts of the system (bus)

This is exacerbated by differences in implementations between generations of processors, between processors from different manufacturers,
within the same generation of the same manufacturer.
On top of that, we are working in a general purpose OS, with different processes and kernel code mutually influencing each other,
which adds to the chaos.

In short, despite its great speeds, x64 is not so good for real-time tasks :)

We can talk about probabilities: total cycle execution time, average iteration execution time, bottom and top estimates.

C++ has chrono::high_resolution_timer. But on different compilers, different OSes and processors its accuracy varies by three orders of magnitude.
The most accurate value I have seen on Linux / Intel(R) Core(TM) i5-4690K CPU @ 3.50GHz / gcc 8.4, with an accuracy of 30 nanoseconds.
On an Intel Core i5 480M @ 2.67GHz / Windows 7 / MSVC 2017, accuracy is on the order of 4000 nanoseconds.
On an Intel Atom x7-Z8750 @ 1.6GHz / Windows 10 / MSVC 2017, the accuracy is 320 nanoseconds.

On x64, timing measurements can be done with Timestamp Counter (TSC), an MSR register available on all x64 generations.
On recent generations, it contains the number of fixed frequency timer clock cycles since the processor reset.
The meaning of this value differs for different generations and processor manufacturers.
The guarantee of the fixed frequency timer can be checked by the presence of the constant_tsc flag in /proc/cpuinfo.
Timer reading is performed by RDTSC/RDTSCP instructions, which in turn has a price in clock cycles.
Timer reads themselves affect execution: they change the state of the pipeline, they are the subject of unscheduled execution,
affect the cache (because we put the read data somewhere).

The methodology of timing measurements is as follows:
1. Calculate the price of RDTSC instruction in clock cycles at the Ongoing* load:
- load the converter with a cycle from this instruction, calculate the average number of clock cycles per instruction
2. Get (empirically**) the current timer frequency:
- measure several times how many clock cycles occur in one second***
3. Determine the price in nanoseconds of one clock, based on the data from 1 and 2.
4. Sample the measured section with RDTSC instructions, taking into account the price of RDTSC itself
5. Interpret multiple measurements.
* When load changes, RDTSC price may float - due to floating processor frequency, converter state.
** Accurate and universal obtaining of timer frequency is quite difficult, both because of different meaning of this value itself
on different processors, as well as due to the difference in the required input data and formulas.
https://stackoverflow.com/questions/42189976/calculate-system-time-using-rdtsc
That's why we cut a corner here, go into empiricism, but get quite plausible values.
*** And here we rely on the implementation of the sleep function in the OS/compiler, which is very noisy.

Interpretation of the measurements:
- zero time value means an out-of-order execution of two RDTSCs in a row:
the execution of the section being measured was blocked and the processor had to execute the second measurement instead.
That is, this code was not actually executed for zero time, but on the contrary longer than usual.
- Big bursts are context switching and waiting for another thread to work off its time quantum.
Besides, it may be thread migrations between cores - both the time associated with it,
as well as the timer mis-synchronization between cores (there is no guarantee of their synchronization).
- Smaller spikes are waiting for data transfer over the bus during cache misses
- the other values more or less correspond to the truth.

As we can see, the methodology is imperfect.
Strong noise prevents us from obtaining accurate timings, but it is still possible to obtain at least qualitative characteristics of the
of code execution and calculate probability distributions of times.

x86 processors have a built-in performance monitoring unit, the Performance Monitoring Unit (PMU),
among which is a frequency-independent register (describe Coreclock register)

REAL-TIME TASKS IN GENERAL-PURPOSE OPERATING SYSTEMS

The best OSDS is single-tasking. Any context switching introduces a delay in execution; the cost of this delay can be immeasurable.
In MS DOS, the context is switched only on interrupts; in RT-11, even at the completion of asynchronous I/O.
Modern OSDSs have added timers, synchronization primitives.
This is the maximum service that the OSDS can allow.
Attempts to add multitasking to the OSDS deteriorate its characteristics.

There is a well-known realtime patch for Linux, it really increases the predictability of the scheduler, reduces the number of context switches
and reduces the latency of the SW process. This is suitable for many microsecond precision tasks.
A visualization of context switch jitter in interrupt handling: https://habr.com/ru/post/562636/.

The most radical way to simply and cheaply get OSRV out of regular Linux is to allocate separate cores to SW tasks (thread affinity).
All interrupt handling and system processes are moved to other kernels.
This eliminates the very possibility of context switching for SW processes (except for some mandatory interrupts).
https://access.redhat.com/documentation/en-us/red_hat_enterprise_linux_for_real_time/8/html/tuning_guide/chap-general_system_tuning


HIGH-SPEED NETWORKS

To reduce latency (minimize response time) such hack as Kernel Bypass is used:
Processing of network stream bypassing the kernel, in userland. In particular, this is used in SolarFlare network cards (their term is OnLoading).
The whole TCP/IP stack is compiled in a library running in user mode.
Interrupts from the hardware are of course there, and they are handled in the kernel; but the network filter (iptables) and all unnecessary things are excluded from the packet adventures in the kernel.
So unnecessary kernel/user context switches, intra-kernel locks (which for example can be clamped for our process by something else) are removed,
buffer copies, the possibility of re-planning an overextended handler in the kernel, and so on.
This is usually combined with affinization of SW tasks (see above) on a dedicated kernel.
The library uses spinlocks as locks as much as possible.
For a programmer it looks like ordinary sockets.

Bonding to describe.

To increase bandwidth is used RDMA - remote direct access to memory (another machine in the network).
In this case, copying to computer memory is done directly from the hardware receive buffer of the network card,
bypassing processing by the central processor, bypassing the OS as such (in fact, without interrupt generation).
Low latency is a bonus here, but it is most effective for thick constant data flow (for example, in clustered DBMS).
An example of hardware is Infiniband, libibverbs library.
It is used in interlinked clusters, clustered DBMSs.

The common place of high-speed networks is zero copying of buffers (zero copy).
On a gigabyte stream, buffer copying time *already* halves throughput.
On a 100GB stream, do the math yourself.
That's why all calculations and data manipulations should be performed directly in transmit/receive buffers.

SHADOW BUFFERING (SHADOW BUFFERING/DOUBLE BUFFERING)

Shadow buffering is used to achieve high throughput of data transfer (to the network, when dumping to disk):
- several buffers are allocated, only one of them is active (ready for transmission) at one moment of time
- inactive buffers are filled with data in separate threads
- meanwhile, data are transferred from the active buffer
- when the transfer from the active buffer is completed, it is marked as inactive; the next buffer ready for operation is selected.

This is a special case of the ring buffer idea.
