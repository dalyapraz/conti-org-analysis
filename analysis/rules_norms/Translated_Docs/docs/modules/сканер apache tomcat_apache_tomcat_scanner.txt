APACHE TOMCAT/CGI-BIN VULNERABILITY SCANNER (CVE-2019-0232)
Technical Specifications


PURPOSE

Need to develop a domain scanner with CVE-2019-0232 vulnerability.


EXPLOIT

The essence of the exploit is incorrect escaping of the command line of the Apache Tomcat server cgi-bin script.
Because of this, remote code execution is possible.
If you make a request like this

http://localhost/cgi-bin/hello.bat?&whoami

the result will be cmd.exe /c "whoami"

Details are described in the links:
https://blog.trendmicro.com/trendlabs-security-intelligence/uncovering-cve-2019-0232-a-remote-code-execution-vulnerability-in-apache-tomcat/
https://wwws.nightwatchcybersecurity.com/2019/04/30/remote-code-execution-rce-in-cgi-servlet-apache-tomcat-on-windows-cve-2019-0232/


REALIZATION

We will use the https://www.shodan.io/search?query=apache+tomcat service as the scanning source
To access shodan.io we need an account. But in idea it is safe even when implemented as a module to the bot.
It is possible to pre-download the necessary data in .csv format and load them separately into bots via configs.

The result of scanning is a list of records with fields:
- URL of the original request (http://domain.com/cgi-bin/script.bat?&dir for example)
- full HTTP response to the request, including all HTTP headers and response body
- URL of whoami request (http://domain.com/cgi-bin/script.bat?&whoami)
- full HTTP-response to whoami request (see below), including all HTTP-headers and response body

The scanning algorithm is as follows:
1. Pair the shodan.io response to this request, taking into account paging. Get site names from the response.
EITHER work from a .csv or .json file exported from the service (this is a paid service on the service).
2. take the next domain from the scanner output.
3. If there is a server software version in the headers, make sure it is less than 7.0.94, 8.5.40 and 9.0.19 respectively.
If the version is greater than or equal to, skip the domain and move on to the next one.
If the version is less or absent, continue.
4. make queries

http://domain.com/cgi-bin/hello.bat?&dir
http://domain.com/cgi-bin/hello.bat?&dir
http://domain.com/cgi-bin/test.bat?&dir
http://domain.com/cgi-bin/info.bat?&dir
http://domain.com/cgi-bin/0.bat?&dir
http://domain.com/cgi-bin/1.bat?&dir

We continue the search either until we have searched all script variants, or until we get the answer we need.
As you can see, file names vary - it is better to put these names into an array in the program and work by brute force.

The names of the scripts for testing:

 hello.bat
 helloworld.bat
 test.bat
 index.bat
 info.bat
 sysinfo.bat
 run.bat
 tomcat.bat
 0.bat
 1.bat
 2.bat
 ...
 10.bat

In addition, for each script name, we need to add digits from 0 to 9 to its name.
That is, for the name hello.bat we will try the following options:
hello.bat
hello0.bat
hello1.bat
..
hello9.bat

IMPORTANT: see why the metasploit works without script name matching!
https://www.exploit-db.com/exploits/47073
ANSWER: because the script name is set externally, i.e. we don't really know the script.

4. If code 200 was received in response, you should analyze the response for vulnerabilities:
- for the dir command, there are always stable patterns in the response
mm.dd.YYYYY HH:mm <DIR> .
mm.dd.YYYYYY HH:mm <DIR> .

If this pattern is present in the response:
- query http://domain.com/cgi-bin/<script>.bat?&whoami
(here script is the name of the script for which we managed to get a response 200 with the pattern we are looking for).
- add requests and responses to the resulting list.


The scanner should be designed as follows:
* it can be a script in a common scripting language (e.g., python or PowerShell)
* the script should log in the current directory, named $(script).log, where $(script) is the name of the script itself.
* the log should have date-time stamps for each event
* logging can be basic or advanced - switchable by command line option
* the following events should be added to the log:
- program startup, with its version
- detection of a domain with vulnerability:
  - domain
  - software version
  - complete HTTP requests and responses
every H minutes (as an option - search iterations) output statistics:
- % of domains scanned, % of vulnerable domains found
- in case of advanced logging, log every HTTP request and response (even if they are not successful).
