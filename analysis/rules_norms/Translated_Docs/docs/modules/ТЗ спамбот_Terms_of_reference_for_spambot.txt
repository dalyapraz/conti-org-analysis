Technical Specifications
SPAM BOT

OBJECTIVE
To develop a spam bot that uses host computer resources to send spam.

MODES OF OPERATION
There are two main modes:
- contact collection: only collect addresses and contacts from local mailboxes and send them to the server
- collection and distribution: the same + spamming.

The input data for the bot is
- email text
- attachments
- mailing list
- sending vectors (lists of SMTP servers, local mail client)

The first two items (mail and attachments) are received by the bot from the server.
Mailing lists and sending vectors can be both internal and external:
- internal - these are obtained from local computer resources
- external - set from the server.

Additional modes:
- working by locally mined mailing list, or by the one received from the server;
- sending through local mail clients or through SMTP-servers specified by the server.

MAIL RANDOMIZATION
The bot must support randomization of emails and attachments, by using macros in the email body and attachment names.
Macros include:
- dictionaries (selecting a random word from the dictionary);
- generation of a random number;
- generation of a random string.

(Thinking out loud) there are two ways:
- randomization on the server: pros in simplifying the client, cons in increasing the load on the server
- randomization on the client: the opposite is true.

REQUIREMENTS
- communication with the command server to receive the mail, attachments, list of addresses for mailing and list of SMTP-servers.
- design as a statically linked .exe
- complying with the requirements of "code and assembly design" for protection against analysis and detections
- the program must not attempt to obtain its own path
- the program must not restart itself
- the program must work with any rights - including SYSTEM rights!
- the program must work in any session - both interactive (>0) and system (session 0).
Check psexec -d -s program.exe
- HTTP exchange implementation - WinInet/WinHTTP by default. However, we need to make simple wrappers for HTTP requests,
to easily and simply replace the implementation

DEVELOPMENT STEPS
1. only collecting addresses from Microsoft Outlook
2. mailing to the addresses collected in step 1 via Microsoft Outlook
3. obtaining addresses for mailing from the server, mailing via Microsoft Outlook
4. in addition, mailing to the list of SMTP servers obtained from the server
5. collection from GMail webmail
6. mailing via GMail webmail
7, 8 + Yahoo Mail


PROTOCOL.
The protocol is plain HTTPS.
The HTTP server MUST provide a Content-Length header for each response.
Request and response bodies must be encrypted with a simple XOR, using the URI as the gamma (encryption key).
That is, for a GET /1/2/sv request body, the encryption key would be the string /1/2/sv

-Strikeout-.
Communication with the command server occurs through .bazar domains (or another Emercoin DNS domain).
There may be multiple domains; for the initial connection to the server, the domains are randomized
until the first successful response. The programmer must assume that all domains lead to the same server.

After resolving an IP address from an Emercoin domain, TWO addresses should be sampled:
- directly obtained from the DNS response
- resolve the IP address by performing an XOR 254 operation on each octet. For example,
  124.245.101.251 (obtained from DNS response) -> 130.11.155.5

Since DNS information is available to everyone, this way we protect against abuses of DNS-recorded addresses.

The code of the PowerShell script ipxor.ps1:
$ip = read-host -prompt "Enter IP";
write-host $ip;
$newip = '';
($ip.split('.') | foreach {
    $octet = [byte] ( $_)
    $octet = $octet -bxor 254;
    $newip = -join($newip,'.',$octet);
}
)
write-host $newip;
-End of crossed-out-

Communication with the command server takes place through a hardwired list of IP addresses.
The first address responding to port 443 via HTTPS is selected by brute force.
Further work is done with it.

When establishing a connection, in case of timeout or connection failure, you should repeat the attempt once.
Then you should proceed to the search of addresses until the first successful response.

-Delayed-.
When tapping, keep in mind that the duality of addresses due to XOR encryption can lead to
random coincidences: you may get a response from a server that has no idea.
-End of crossed out-


The request URI is

/id_mailout/id_bot/request

where
- id_mailout - mailout identifier - uint64_t
- id_bot - bot identifier - string; by default it is GUID in the format CBD27893-F9EB-465C-AAF6-D620ED24C8C9,
but it can also be an arbitrary string with valid characters a-zA-Z0-9_-.

Both parameters can be passed to the bot externally, via command line arguments or environment variables.
The default response format is plain text with a [\r]\n line separator

By default, any response code other than HTTP 200 or HTTP 102 is an error (including codes 204, 30*).
HTTP 102 "Processing" means that the server cannot respond right now, and should repeat the request after some time.
In this case, the bot should do cyclic polling at 30 second intervals until a non-102 code is received.
In the case of an error code (not 200 or 102), the bot should retry five times at 5-second intervals, until the first success.
If there is no success, the bot records the error and terminates.
If the bot is unable to parse and interpret significant fields from the server response, the bot will log an error and terminate.

1. Give parameters: mail interval, mailing list type, send type, number of attachments, control box
GET /<idm>/<idb>/st

Response:
<interval>[\r]\n // the number of seconds between sending two consecutive emails. 0 means no interval. The default is 0.
<mailout_type>[\r]\n // bitmask: 1 - all addresses from local desktop client;
                               // 2 - all addresses from local webmail accounts;
                               // 4 - get the mailing list from the server;
                               // other values (bits) - error
<send_type>[\r]\n // bitmask: 1 - sending via desktop local mail client;
                               // 2 - sending via local webmail accounts;
                               // 4 - sending through the list of SMTP servers received from the server.
                               // no bits are set - only contact collection
                               // other values (bits) - error
<attach_num>[\r]\n // number of email attachments: from 0 to 10 (other values - error)
<rcpt_every>[\r]\n // one letter for the specified number of recipients; additional recipients are specified in Bcc:
<random_every>[\r]\n // randomize every Hth email
<control_mailbox>[\r]\n // control mailbox
<validate_emails>[\r]\n // whether to validate email addresses after collection: 0 - no, 1 - yes

2. give the subject
GET /<idm>/<idb>/sb/<number>

In response - email subject, one per line (you can get many subjects at once).
<number> is the requested number of topics.
In this request and in request #3, the <number> parameter is used to minimize the number of requests to the server.
For example, to get 1000 e-mails at one time.

3. give the email
GET /<idm>/<idb>/m/<number>

<number> - the requested number of emails.
In response - the text of the email as it should be sent.
It is possible to specify here - since the mail can be in a multipart container.
If all mail clients allow you to specify a letter in raw form (with all container separators, etc.),
then in response to this request we get a fully assembled raw body.
Otherwise, we need subqueries to get all parts of a multipart message.

4. give attachment no.
GET /<idm>/<idb>/a/<#>

Here # is the number of the attachment from 1 to 10.
The response must contain a Content-Disposition header unambiguously identifying the name and MIME type of the attachment.
Absence of this header is an error.
The content of the response is interpreted by HTTP response headers (container, compression headers, etc.).

5. give the mailing list
GET /<idm>/<idb>/l

The response contains a simple listing of email addresses to which to send the email.
One address per line.

6. give the sending server
GET /<idm>/<idb>/sv

The response simply lists the SMTP servers through which to send the email, one per line.
The format of the string is:
[<proto>:]<IP-address>:<port>:<user>:<password>[:field=value[:field=value...]]]
The contents of the proto field indicates the need to work over an SSL channel.
Specify.

The field=value parameters are used to send additional fields, such as DNS records, RSA keys, selectors, etc.
The number and purpose of the fields is interpreted by the specific server and bot implementation.


7. get contacts
POST /<idm>/<idb>/c

The content of the request body is plain text (WITHOUT multipart or form-urlencoded!)
one contact record per line, UTF-8 encoded.
The contact record has the following format:
field=value;field=value;...;field=value\r\n
All value values are urlencoded.
The following values are supported for field:

email
firstname
midname
lastname
country
state
city
zip
street
phone
url

Any unknown values must be stored by the server in the database! There should be a mechanism for this,
that allows saving extended contact fields.
At the same time, you need to eliminate denials of service by sending gigantic queries.

If the sending volume exceeds the threshold (set by a compilation constant), the bot must split the sending into a corresponding number of requests.
into a corresponding number of requests. By default, the request limit is 2M.

8. take the information
POST /<idm>/<idb>/info

The content of the request body is plain text (WITHOUT multipart or form-urlencoded!)
of the following form:

os: <OS name and version>
arch: <machine architecture x86/x64>
hostname: <computer name>
username: <username>
domain: <domain or workgroup>
tz: <temporary zone>
locale: <local>
ip: <external IP address>
client: <name and version of local mail client, if found. There may be many such strings>
webmail: <URL and email client account, if found. There may be many such strings>
me: <email and local mail account contacts. There may be many such strings>

The bot can send more information in field:value format - this request may be subject to expansion.
The server implementation should take this into account and be prepared to store unknown data.

9. take statistics

POST /<idm>/<idb>/info

The content of the request body is plain text (WITHOUT multipart or form-urlencoded!)
of the following form:

sent: <total emails sent, pieces>
addresses: <total addressees>

The bot can send many such requests during one session to reflect the progress of mailing.
The server is focused on the last request received from the bot.


OPERATING ALGORITHM

The principles of establishing communication with the server are described in the PROTOCOL section.

After starting the bot should:
1. establish communication with the server - find a live domain, make a GET / HTTP/1.1 request and make sure that the response code is 200,
that the response code is 200.
2. collect information about the system and send it to the server. Check the server's response to make sure
that it is "our" server, otherwise terminate.
3. request the operating modes from the server
4. if specified in the modes, collect addresses from local mail clients, validate them and send them to the server.
5. if specified in the modes, send out the mails.
5.1. when mailing, reflect progress by sending statistics periodically (once every 1000 addresses or 1 minute, whichever comes first)
6. self-delete from disk (if specified in build options).
