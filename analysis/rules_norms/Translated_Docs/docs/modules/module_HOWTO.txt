MODULE DEVELOPMENT RULES

A module is a DLL that will work in the left arbitrary process.
There will be a 64-bit module on a 64-bit system and a 32-bit module on a 32-bit system.
A module is not just a C++ program. A module has some limitations due to the context in which it works:
1. We need to minimize the use of Visual C++ Runtime. The malloc, memcpy, str* functions should be avoided. WinAPI already has everything you need - instead of malloc - HeapAlloc, instead of memcpy - CopyMemory, instead of strstr - StrStr (shlwapi.h/.lib).
1.1 STL should be avoided in general. STL does not work with std::mutex. The rest is at your own risk!
1.2. Some STL chips give additional prologue to the code. For std::mutex it weighs 200k. It is not visible in your code, but it is there and will behave strangely in the context of execution!
1.3 Not following points 1 and 1.1 can cause VERY strange behavior.
2. linking of rantime and third-party libraries to the module should be static (flag /MT).
3. If third-party libraries are used, they should allow replacing malloc/realloc/free memory management functions (instead of them you will slip your VirtualAlloc wrappers in there). They should also be statically linked with everything you need (so that you don't try to load some dll that Windows doesn't have by default).
4. The module can run both as the current interactive user and as the local system (nt authority\SYSTEM), as a non-interactive process (as a service). For non-interactive, google Session 0 isolation/Services isolation in session 0.
5. Module code should not try to get the path to itself, because in real conditions it will run in memory without saving to disk.
The developer must take this into account and ensure that the module works in all modes. Ignoring or not knowing these points is usually the reason why you have a module running locally and not in production.
Also, there are additional requirements related to security:
1. Temporary files are prohibited. By default, writing to any files and registry is prohibited. Each case of such writing must be specified.
2. If possible, the module should not have a state (i.e., it should not need to save anything to the registry/file when stopped and then read from it when started).
3. If the module runs in multiple processes, the best way to communicate between them (in descending order - remember that temporary files are forbidden):
a. Direct memory read/write (ReadProcessMemory/WriteProcessMemory)
b. Named pipes


MODULE DESIGN
1. You must have a Microsoft Visual Studio project of version 2015 or higher.
2. The Visual Studio project must be configured as follows:
* For ALL build profiles:
- Output directory: $(SolutionDir)Bin\$(PlatformTarget)\$(Configuration)\.
- Intermediate directory: $(SolutionDir)\obj\$(Platform)\$(Configuration)\$(ProjectName)\.
- Multiprocessor Compilation: Yes
* Release Profile:
- Debugging Information Format (C/C++ code creation): no
- Create debugging information (linker/debugging): no
3. Obfuscate strings with Andrivet library (attached, see _STR() macro)
4. Obfuscate system calls with GetApi.h library. Be careful, obfuscation of syscalls may cause crashes.
5. The module should have two versions - x32- and x64-bit.
6. In the combat build, lines should be obfuscated to the maximum and all debugging output should be disabled.
7. The module must have a debug version. Debug output should be output to c:/temp/modulename.log (the path to the log is configurable in the macro).
Each log entry must contain a timestamp accurate to the second.
8. There should be a config.h file in the project (the name is not important, the essence is important - all global settings - paths, macros-switches for conditional compilation, etc.).
9. The module should work on all modern versions of Windows.
   The minimum supported version of Windows is Windows XP (if it is impossible - Windows Vista).
10. Additionally, the file notelemetry.obj (https://stackoverflow.com/questions/37761768/how-to-prevent-visual-studio-2015-update-2-to-add-telemetry-main-invoke-trigger) must be added to the layout.


ASSEMBLY NAME

The assembly must have a directory structure:

 modulename/Release_logged/x86
 modulename/Release_logged/x64
 modulename/Release_nologs/x86
 modulename/Release_nologs/x64

Release_logged contains the version with logging and without obfuscation (but it is still a Release build profile!
i.e. it should have all optimizations enabled, debugging symbols disabled, etc.).
In release_nologs - version without log, with full obfuscation.
The name of the directory with the build should contain the name of the module and the date (for example cookies.22.04.2019).
If there are several builds of the same module on the specified date, a unique suffix should be added (e.g. cookies.22.04.2019.2).

This is necessary for normal operation of the module - so that testers and administrators can distinguish versions of builds,
so that it is possible to rollback or upgrade, and also to provide the required build by its date.


MODULE TESTING

The module should be tested in the following modes:
1. On Windows 7, 8.1, 10 operating systems of both 32- and 64-bit versions. The OS should NOT have programs that install MSVC++ runtime library packages (i.e., we should install a bare OS and the minimum necessary for testing).
2. as an interactive user (the one you are logged in as).
3. as a non-administrator
4. as System (download the pstools package, run the module as follows:
psexec -d -s runmodule.exe
this is running the module as a non-interactive SYSTEM user.
Only after passing internal tests you can pass the module to the tester.


INTERACTION WITH HIGHER-LEVEL LOGIC

The module receives the necessary information from higher-level logic. ClientID, group group tag, external IP address will be passed through the Start function in the ParentInfo parameter
The module receives configs through the Control function (if there are several configs, each of them corresponds to its own value of the CtlArg parameter).
The body of the configs and their length will be passed through the CtlArg and CtlArgLen parameters. other parameters are not used.
For offline tests, you should write a demo launcher that will load the compiled module via loadLIbrary. The pseudo-code is very roughly like this:
main() {
  LoadLibrary("module.dll");
  Start = GetProcAddress(module, "Start");
  Handle = Start(...);
  If(!handle) return 1;
  Control("config", "config body");
  While(true) sleep(1000);
  Release(handle);
}
Thus, the following scheme:
1. demo launcher loads a module (DLL) to itself and calls start and control functions to pass configs, external ip address, clientid and group to it.
2. code inside the module that does useful work.

The execution flow is as follows:
* The Start() function initializes the job, starts the main worker thread in the background, and returns a module descriptor (usually the address of the Start function)
* The main thread can wait until the necessary Control() calls with configs arrive and only then start working.
  It depends on the purpose of the module itself - there may be no need to wait.
* Control() calls are usually used to pass configs or control actions (change of operating modes) to the module.
  These calls must be handled as interrupts. These calls take place in a separate thread, so thread safety should be ensured for everything,
  that is shared with the main thread of execution.

MODULE API
The module exports the following functions: Start, Control, Release, FreeBuffer [*1]
All functions have the calling convention stdcall and are exported by name.
The Start function has the following prototype:
PVOID Start(
	LPCSTR ModuleName,
	LPCBYTE Arg,
	SIZE_T ArgLen,
	LPSTR ResultInfo,
	const ParentInfo* pParentData,
	PVOID EventCallback,
	PVOID EventCallbackContext,
	PVOID Reserved1);
The function is called when the module is started by the above logic.
ModuleName - module name
Arg - argument of the start command
ArgLen - size of CtlArg parameter in bytes
ResultInfo - buffer for the resulting ctl string. The buffer has a fixed length of 1024 bytes.
pParentData - information about the above logic, controlled by the module's config.
EventCallback - pointer to the logging function (see below)
EventCallbackContext - logging context (see below)
In case of success, the function returns a descriptor to be used when calling Control and Release functions (you can pass the address of the Start function itself - it is quite unique within the OS). In case of failure, the function returns null.

typedef struct ParentInfo {
	CHAR ParentID[256];
	CHAR ParentGroup[64]; CHAR SelfIP[64]
	CHAR SelfIP[64];
	LPCWSTR ParentFiles;
} ;
ParentID - the full client ID of the above logic
ParentGroup - grouping tag of the above logic
SelfIP - external IP address
ParentFiles - not used

6.2 The Control function has the following prototype
BOOL Control (
	PVOID ModuleHandle,
	LPCSTR Ctl,
	LPCBYTE CtlArg,
	SIZE_T CtlArgLen,
	LPSTR CtlResultInfo,
	PVOID* ppOutData,
	PDWORD pOutDataSize,
	LPCSTR pOutDataTag,
	PVOID Reserved1);

ModuleHandle - module descriptor returned by the Start function.
Ctl - string containing the description of the control signal to the module
CtlArg - ctl argument to the module (signal body)
CtlArgLen - size of the CtlArg parameter in bytes
ResultInfo - buffer for the resulting ctl string. The buffer has a fixed length of 1024 bytes.
ppOutData - pointer to the variable where the pointer to the buffer with ctl output data (ctl_OutData) will be saved.
pOutDataSize - pointer to the variable into which the data size in the buffer with output data will be saved ctl
pOutDataTag - buffer for the auxiliary tag that will be sent to the server. The buffer has a fixed length of 128 bytes
The function returns TRUE in case of success, otherwise the function returns FALSE. In case of success, if the value of *ppOutData after the call is not equal to zero, this buffer must be freed via the FreeBuffer function.
The usual purpose of this function is to pass the configuration to the module.

The Release function has the following prototype
VOID Release (
	PVOID ModuleHandle);
The function implements the complete termination of the module's operation. Its tasks include deleting all resources used during the module operation.

The FreeBuffer function has the following prototype
VOID FreeBuffer (
	PVOID pMemory);
The function frees the buffer allocated inside the Control function (parameter ppOutData).

* Changes in naming of functions in the export
1) the old version continues to work, but antiviruses often flag the module by these functions.
2) The module exports any minimum of 4 functions by name, the function names are reported to the administrator as
CheckFuncStr=Start, GetLength=Control, SetHeigth=Release, Reload=FreeBuffer.
Functions can be changed during cleanup, but again you will need to inform the administrator.
3) The module exports functions by ordinal, or by name, but necessarily in the following order: 1.Start, 2.Control, 3.FreeBuffer 4.Release
Example def file for exporting by ordinal
EXPORTS
@1 = Start_
@2 = Control_
@3 = FreeBuffer_
@4 = Release_
Example def file for export by name:
CheckFuncStr = Start_
GetLength = Control_
Reload = FreeBuffer_
SetHeigth = Release_

RULES FOR LOGGING EVENTS TO THE ADMIN

The penultimate two parameters of the Start function are for logging events to the admin area.
EventCallback is a pointer to a function defined as:

typedef VOID (__stdcall *pEventCallback)(
	PVOID ModuleHandle,
	LPCSTR EventName,
	LPCSTR EventInfo,
	PVOID pOutData,
	DWORD OutDataSize,
	LPCSTR pOutDataTag,
	PVOID Context);

where:
	ModuleHandle - module descriptor (usually a pointer to the Start function)
	EventName - event type
	EventInfo - event content
	pOutData - additional event data
	OutDataSize - length of additional data
	pOutDataTag - event tag
	Context - pass here the EventCallbackContext received in Start

Define a function like this in your code
void SendEvent(char* name, char* value) {
	pEventCallback callback = (pEventCallback)EvCallback;
	if (callback) {
		debug_printf("SendEvent(%s, %s)\r\n", name, value);
		callback(Start, name, value, NULL, 0, tag, EvCallbackContext);
	}
}
where
tag - event tag (needed for filtering in logs; it is usually the name of a module, e.g. "module1")
name - event source (many events can be grouped by their source - the subsystem in the code where they are generated. For example, "sql")
value - event content (an arbitrary string. For example, "module1 build 01 Jan, 20xx 11:22:25 started ok")

The developer defines the source of the event himself.
The source must consist of 4 characters. For example, you can define types DEBG for debugging, VERS for sending your own version, PING for sending heartbeat to the admin, etc.
A source that identifies the subsystem of your module from which the event originates. For example, a module has a file subsystem and a network subsystem - the tags "file" and "net" are used for them.
In the past, you had to put mutex on the logging, because there was no protection and verification in the above logic, and it was thread-safe. Then they put the mutex on the other side, and everything seems to work.
But - if anything - if you have problems - look first of all at multithreading, and secondly at call convention (_stdcall).

!!!!!!!!!!!
!!!IMPORTANT!!!
!!!!!!!!!!!

Events from the module should be few and important. It is not allowed to flood with debugging.
i.e. you can set logging in the module for debugging, but then you must remove debugging events when you release the build.

Sending an event is a blocking operation.

You should realize that sending an event to the parent is not instantaneous. The parent receives events by periodic polling
through shared memory. Therefore, although an event may be sent, it may not yet be received by the parent.
This is especially true for events sent before the process terminates - they may be lost for the reason described.

If a module is non-resident (it runs once and then shuts down), it must send an event when it terminates
EventName = WantRelease
EventInfo = NULL
Upon receipt of this event, the module's medium will unload it from memory.


TYPICAL ERRORS

1) sending an event (message), most often with the module version and date from the Start function.
This should NOT be done.
The correct way is as follows:
- Start performed initial checks, initializations, started the main thread and terminated
- the main thread already starts spamming telemetry in events

2) botid, group and ip are passed to the Start function.
These string values must be copied in the Start function, because when you exit the function, the memory of the strings is released and the strings become invalid.
