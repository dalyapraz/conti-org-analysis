1. INJECTION INTO A PROCESS

Injection into a process is needed to execute one's code in another's process of the same user, in a situation where there is already access to the system.
Classic targets are browsers (intercepting and replacing traffic to realize MITM attack), messengers, email clients, etc.
The sequence of calls is approximately as follows:

OpenProcess - open someone else's process
VirtualAlloc - allocate memory in it
VirtualProtect - allow execution of this memory
WriteProcessMemory - write your shell code into this memory
CreateRemoteThread - start the shell code as a new thread

If the shell-code is short and written with relative addressing, this is usually enough.
But this is a rare case, such code is usually written in assembly language.
More often you need to execute large pieces of logic in the form of a dll.
To run a dll in someone else's process, this dll must be copied into someone else's memory,
and customize it - write all the imports, set up the addresses of the phases, etc.
In normal life this is done by the LoadLibrary function.
But this is too conspicuous function, which also accesses the disk.
Completely diskless injection from memory is implemented by two libraries:

https://github.com/stephenfewer/ReflectiveDLLInjection
https://github.com/dismantl/ImprovedReflectiveDLLInjection

The second library is just an improvement of the first one:
- cross-bit injection works (32->64, 64->32, 32->32, 64->64)

Basically, the implementation has an injectable loader that uses only relative addressing.
The injection proceeds as follows:
- a loader is written into one piece of memory
- a dll is written into another piece of memory
- control is passed to the loader
- loader searches the import table for LoadLibrary and GetProcAddress functions by hashes of their names
- loader searches for the dll in memory and configures it
- after customization the loader transfers control to DllMain

This technique can be improved by converting the injected loader into an external loader:
- it will be executed from the attacking process, not from inside the attacked process
- All memory handling functions in it should be replaced by WriteProcessMemory/ReadProcessMemory.
No sources of the ready implementation have been found in the network yet, but there is such an implementation in nature.

2. FUNCTION INTERCEPTION (HOOKS)

A technique known for decades:
- in the prologue of most system functions there are several nops (up to 5) provided
specifically for this purpose
- the function's prologue is replaced by jmp on our handler
- the handler calls the original function by the prologue offset after nop (or not - depends on the logic).

For everything to work without errors, the signature of our handler function must completely match the data types,
return types and call convention.

If there is no special prologue, the hook can still be placed - not always. You have to look at the contents of the prologue.
Simply jmp instructions to be overwritten should be copied and moved to the body of a new handler - but there is a creative approach,
and there are no universal recipes.

The Microsoft compiler has a switch: Project properties -> C/C++ -> Code creation -> Create patchable image (patchable image) specifically for the purpose of creating a patchable image.
(patchable image) specifically for the purpose of generating empty function prologues.

This hook is also called a springboard, detour or inline-hook.

You may see variations of trampolines in the article "Trampolines In x64" https://www.ragestorm.net/blogs/?p=107.

This article https://www.malwaretech.com/2013/10/ring3-ring0-rootkit-hook-detection-22.html

3. PROCESS HOLLOWING

Process Hollowing (aka RunPE) is a Windows implementation of the Unix analog of the fork()/exec() syscall pair. That is, starting a process
and replacing its text with another executable, setting it up in process memory, and then running it from the entry point.
In Windows, instead of fork()/exec(), CreateProcess() is used, which does both fork() and exec() at once.
I.e. the process text is replaced, which we cannot control.
That's why the implementation of text replacement, import settings and relocations must be done by yourself, which is quite complicated.
and essentially duplicates the Windows process boot loader.

The advantages of this trick are that:
- our code looks like another process
- the other process can be a trusted OS process (explorer, svchost),
added to UAC, firewall and antivirus exceptions.

The implementation with a detailed description is available at
https://github.com/m0n0ph1/Process-Hollowing

Antiviruses recognize process hollowing by comparing the process text in memory and on disk.

VBScript implementation, for use in macro-droppers:
https://github.com/itm4n/VBA-RunPE

4. PROCESS DOPPELGäNGING

The purpose of the technique is the same - to hide the name of the real running process.
The essence of the execution is different.

- An NTFS transaction is opened to write to some executable file. As usual, it will be
one of the trusted processes - svchost, explorer, etc.
- the .exe body is replaced with our text
- the process is started
- the transaction is rolled back. No physical writing to the file body takes place.

For all other processes it looks as if the original file is launched.
But the thread that opened the transaction sees new text in the file being launched.

The technique has become obsolete, because since some version of Windows 10 NTFS transactions seem to have been canceled completely.

5. PEB REPLACEMENT

A lightweight technique for hiding a process post facto.
PEB is Process Environment Block, a structure with basic information about the process, is present in every process.
In particular, it contains the process name and its command line. These can be replaced both from within the process and from outside the process
(suspend process threads with SuspendThread(), replace data with WriteProcessMemory() and resume threads
using ResumeThread()). After that, you can see the changed process name and command line in the process list.

Another interesting trick is to overwrite the .dll export table after loading it into memory.
After the addresses of all the necessary function addresses are obtained, you can overwrite the whole export and thus
make it difficult to identify the process by known/unique function names.

Additionally, the section with resources can be erased if they are not used (or after they have been loaded).
In short, variations with swapping and erasing service tables/areas are limited only by imagination.

6. SHELL CODE

The name comes from the ancient times when injected code was intended to get a remote shell on the attacked machine.
Therefore, the code had to meet the requirements
- be as concise as possible (in order to obfuscate the smaller parts of the attacked process)
- be position-independent (relative addressing only)

Nowadays, shell code refers to any insertion of machine code regardless of the application context.
However, the typical context is still injection into a process, e.g. roughly into a piece of an executable thread (to bypass DEP).
So the above requirements still apply.

Typical shell code for Windows processes does bootstrapping (including to bypass ASLR):
- determines important memory milestones - pointers to TEB and PEB
- gets the address of the import table from PEB
- gets the address of kernel32.dll from the import table
- from kernel32.dll gets the addresses of LoadLibrary, GetProcAddress (usually by name hash, not by the name itself)
- when LoadLibrary and GetProcAddress are available, you can do anything you want.

               II. SECURITY TECHNIQUES


HOW ANTIVIRUSES WORK

AVs get information about what is happening in the system in one of two ways:
- By subscribing to kernel events using Event Trace for Windows (ETW) https://docs.microsoft.com/en-us/windows-hardware/drivers/devtest/event-tracing-for-windows--etw-
- by connecting their minifilter driver in kernel mode (ring0), and thus subscribing to file system activity and similar events (Windows Defender, Eset, WebRoot, McAfee).
- putting their hooks in user mode (ring3) directly into executable processes (Avast, BitDefender, Symantec, TrendMicro) to functions from ntdll.dll, user32.dll, kernel32.dll, etc.
In the first case there is nothing you can do, in the second case the hooks can be removed.

WINDOWS DEFENDER DEVICE

Windows Defender uses a minifilter type driver called WdFilter to subscribe to events on the system.
Intercepted events are:
- process creation and startup
- process image loading
- thread startup
- manipulations with a non-native process - writing to memory and starting a remote thread
- drivers being loaded are verified
- registry operations are intercepted.

Of all the useful things we managed to find in the parsing - a list of conditions under which you can avoid such attention:
- white list of processes (including svchost, werfault and WinDefender processes)
- list of hardened registry keys

EMULATOR DEVICE

AB emulator is intended for additional analysis of possible program behavior during static analysis of binary:
https://findpatent.ru/patent/251/2514141.html
1) the emulator does not execute all system calls, making "stub calls". It may execute some calls that are especially necessary for the program functioning (like memory allocation),
but in general - no;
2) the emulator is more interested in sequences of system calls than in their result;
3) in some ABs the emulator executes all branches in the code regardless of the conditions (according to some evidence of VBA32).
In order to look into all code branches.
4) the emulator is limited by the running time (of the order of tens of seconds, or by the number of executed system calls).

In some ABs, the emulator starts with hooks embedded in your code.

ANTIVIRUS PROTECTION

There are two types of antivirus (hereinafter referred to as AV) detections:
- static (by signature)
- dynamic (by behavior).

The first is easy to fight with, the second is difficult.

In addition, ABs use fuzzy hashing (ssdeep, imphash), neural networks and Bayesian filters to detect previously unknown malware based on the degree of similarity of text or behavior to already known patterns
based on the similarity of text or behavior to known patterns:
https://xakep.ru/2020/09/28/clear-hash/

ABs necessarily put detections on known strings and import functions.

Necessary (but not sufficient) measures to protect against ABs:
- string obfuscation
- obfuscation of imports (i.e. functions imported from third-party .dlls)
- code noise
- code encryption with decryption before execution.

Additional measures to counteract AB:
- disabling AV (with sufficient permissions)
- removal of hooks (only for ABs working in userland)

KHOBE (Kernel HOoks Bypassing Engine) technique is known. There is no code in public, only general information is known.
It is necessary to use race condition (whose?), presenting safe code for analysis at the moment and quickly switch the context right after AB analysis.

For more details, see the section "Cleaning from antivirus detections".

CODE OBFUSCATION

Typically code in .dll form is compressed, encrypted, and packaged into an array (see Hiding data in a code segment below),
allocate memory at runtime, extract and unpack the code, then configure the .dll in memory
(reallocations, imports, and all that). See below about cryptors and packers - this is it.
This trick is known to AB and it does not save from proactive protection.

That's why the thought has moved on:

1. Unpacking of each function just before execution.
Each function defines a prologue and an epilogue that decrypts and encrypts the body of the function, respectively.
Signatures are used to find the function boundary.
An external encoder is required to initially encrypt functions in a file.
Other on-the-fly decoding options are also possible, but all of them are difficult to implement.

2. proprietary JIT interpreters.
For example, https://github.com/jacob-baines/jit_obfuscation_poc.
The idea is clear from the name - we translate one code into another code, which is unknown to antiviruses and inaccessible to automatic reversal and analysis.

STRING OBFUSCATION

Before constexpr appeared, string obfuscation was done by a third-party utility and two-pass assembly:
- strings in code were marked with special markers, usually in the global table of strings
- strings were accessed through the decryption function, by its index in the table.
- the utility would pass through the ready .exe and replace them with ciphertext.

This approach made debugging very complicated and required additional build stages,
making the code unreadable.

constexpr allows you to do string encryption at the build stage, solving all the above problems.
However, this works in Visual Studio no lower than version 2015, requiring C++ standard 14.
Andrivet's ADVObfuscator encryption library is ready:
https://github.com/andrivet/ADVobfuscator
There is also
https://github.com/Snowapril/String-Obfuscator-In-Compile-Time (based on Andrivet)
https://github.com/adamyaxley/Obfuscate
https://github.com/fritzone/obfy
https://github.com/revsic/cpp-obfuscator

Obfuscation works on maximum optimization settings in Visual Studio:
- C/C++/ Optimization: Full Optimization
- C/C++/Optimization / Whole Program Optimization: On
- C/C++/ Optimization / Deploy inline functions: Disable
- C/C++/ Code Creation / Enable line merging: Yes
However, you should try this setting, it affects differently on x64/x86.
- Other optimization settings should be tested too, they may affect.

Developer's comment about obfuscator not working on full optimization for x86 compilation:
"...why string obfuscator doesn't always work.
The matter is in the decrypt method, it turns out when optimization is enabled to unroll substituted functions,
the compiler inserts its body at the place where this method is called, and since this body uses expressions that it can calculate during compilation,
it decrypts the string at compile time.
So it encrypts and decrypts the string at compile time.
It can be fixed by disabling the optimization to unroll substituted functions."

One of the known disadvantages of inline obfuscators is the limitation on string length.
Each additional character in a string is an additional recursion of the compiler during compile-time calculations.
The MSVC2015 stack ends at lengths of about 100 characters.

ENTRY POINT OBFUSCATION

This measure is used to counter AV emulators and manual analysis.
The actual entry point is different from the one declared in PE/ELF headers.
For example, in a .dll there is a fake harmless export (some DllMain, DoTheWork, etc.) that performs some actions for the sake of distraction.
To start the real load, you need to pull the unexported function to an address known only to the launching context.
Another option is to use a DOS jammer. Changing the MZ signature to any other signature in the PE binary file will cause the binary to start in DOS mode.
As a consequence, AB will ignore the true entry point.
Emulators do not work in 16-bit mode; a third-party binary can be started via the 4B interrupt function of DOS.
This can be used in the "chain breaking" technique.

HIDING DATA IN A CODE SEGMENT

ABs are sensitive to an unusually large data section (.data, .rdata) - this is a sign that encrypted code-loading is hidden in it.
Data can be hidden in the text section. Microsoft C++ compiler allows you to do it with the help of such a trick:

#pragma code_seg(push, ".text")
#pragma code_seg(pop)
unsigned char __declspec(allocate(".text")) hiddencode[]={ ... };

On a similar principle you can stack the load in other sections, by pragmas/declspec data_seg, const_seg.
However, the linker may spawn sections with the same names and different access rights, so there is another option:

#pragma comment(linker,"/SECTION:.data,EW")
unsigned char PayloadName0[]={}
#pragma comment(linker,"/SECTION:.rdata,R")
unsigned char PayloadName2[]={}

The counter to this measure on the AB side is the frequency analysis of the code section. The code section has low entropy,
because the number of opcodes is limited and the statistical distribution of symbols in the code has a well-defined structure.
Therefore, the hiding of encrypted and/or packed arrays is quite clearly traceable.
This in turn can be countered by encryption techniques that are weakly entropy-changing - for example,
xor 1 byte (of course, if there is .dll code hidden in such an array. If it contains other data, it won't help).

CRYPTORS AND PACKERS

Robust implementations of packers have been known since at least the mid 90's.
The idea is simple - one .exe is packed inside another .exe, and when executed performs the Baron Munchausen trick of
of pulling the load out of itself.
Of course, this is a great means of hiding code.
Encryption is also added to the packaging.
Packaging can be layered, to make it harder to analyze.
Packers peaked in popularity in the 0s.

The PEId program (discontinued in 2011) was used to determine the packer type.

Nowadays, much smarter cryptors are used.
In addition, the cryptor takes on the functions of bypassing AB emulation, sandbox detection,
sometimes even bypassing UAC and privilege escalation (due to the peculiarities of launching a load, it may be appropriate to assign these functions to the cryptor.
can be appropriately assigned to the cryptor).
Also, besides the trivial pulling of the load from the encrypted array inside itself,
a good cryptor generates plausible import tables, plausible code that confuses AB,
dilutes the entropy of the load, spreads the load randomly across different sections,
generates real resources (strings in localization), in short, pretends to be a real program.
A similar approach is described in https://xss.is/threads/39006/.

In short, it is a protective shell hiding your code.

Of course, cryptors are not omnipotent, and they will not remove behavioral detections.

INTERCEPTION OF OTHER PEOPLE'S THREADS (PROTECTION AGAINST INJECTION)

You can intercept someone else's injection into your process by putting your own handler on the BaseThreadInitThunk() function.
The creation of a new thread starts with it (including those initiated from outside the process).
In this handler, you can decide whether to allow or block the thread to start, based on certain features.
The simplest approach is to start all your threads at once and then block everything else.
If this is unacceptable, you can look at the address and properties of the memory page where the code is launched from.
For an injected thread, this is usually a heap. For a healthy thread it is a text section (.text).

In particular, this is how the Mozilla Firefox browser's injection protection is implemented.

This technique can be successfully counteracted - you can remove the hook on BaseThreadInitThunk from outside the process.
hook removal method described above, after which injection is possible.

Another way is to turn all mitigations (see below) to maximum right after the process start, in particular DEP
and code signing.

CLEANING UP ANTIVIRUS DETECTIONS

The first thing to do before cleaning is to make sure your antivirus is not leaking samples:
- virustotal ALWAYS leaks samples live
- dyncheck drains samples during dynamic behavioral checks. It does not seem to do so with static checks
- In Windows Defender you need to disable the "Send samples" option
- For other ABs you need to find the option to send samples and cloud protection and disable them.

The general cleaning methodology is as follows:
1. Find specific lines of code that are being detected;
2. Replace it.

Point 1 is long and tedious, so we do it this way:
- disable ALL program code with a comment or ifdef, collect it
- AB is shut off
- uncomment half of the code
- AV is silent
- another half of the code
- AB is silent
- half of half of half of half
- AB yells -> we found a section!
Then we reach specific lines by the same dichotomy:
- uncomment a line - screams, comment it - silent.

We take into account the optimizing compiler: the optimizer may throw out a huge block of code,
if it doesn't see the impact of this code on the overall behavior.
For example, if you put return in the middle of a function under test as part of the detection search,
the optimizer may remove both the tail of the function and its beginning from the final binary,
because the optimizer considers that nothing useful and affecting the overall execution occurs in the remaining part of the function.
The optimizer throws out unexpected code fragments by the same principle, which confuses the maps.

As a rule, antivirus detectors are set to:
- the name of the binary file (once the name is flagged, it will give detections)
- Microsoft Visual C++ adds a string with the project name to the binary: Project Properties -> General -> Target Object Name ($(ProjectName) is there by default).
you need to randomize it, or just overwrite it with zeros/spaces in the build post-event right in the binary
- separate system calls (CreateRemoteThread, VirtualProtect, CreateFile, CreateProcess, OpenProcess, registry handling, etc.)
- sequences of system calls (it is silent for individual system calls, and yells for a sequence)
- open lines
- characteristic algorithms (random number generators, (de)encryption, (de)compression)
- high entropy of the binary file (encrypted/archived arrays, including in code sections)

Additionally, a clean file can be detected:
- When downloading it from a site with a low reputation (which has already been complained about suspicious files)
- Alternatively, when a file is downloaded from another country (both the client's address and the site address are affected).
- If there is no hash of the file in the AB database (there are a limited number of launchable files in the world).

Under Linux, don't forget to remove characters and extra lines with strip utility (or even better sstrip).

System calls are either obfuscated by GetApi.h (can be taken directly from carberp/GetApi.h
https://github.com/hzeroo/Carberp/blob/master/source%20-%20absource/pro/all%20source/RemoteCtl/DrClient/GetApi.h)
or, if GetApi does not contain the necessary call, by the following sequence:

HANDLE h = LoadLibrary(_OBFUSCATED("dll.dll"));
void* f = GetProcAddress(h, _OBFUSCATED("funcname"));

here both lines are obfuscated.

Sequences of syscalls are obfuscated in the same way, or syscalls are creatively replaced by analogs.

See above for how to obfuscate strings.

We dilute characteristic algorithms with noise code:
- add/remove volatile to local variables
- change the place of local variable definition in the code (it was in the prologue of a function - put it closer to the place of use, and vice versa)
- add noise code between lines (incrementing a garbage volatile variable, additions, subtractions, other operations)
- such noise code can be formalized into an inline function or a macro: in the debug build the body of the function is disabled, in the combat build - by constexpr's
random code patterns are created.
- entropy is removed by shoving array pieces into different sections and supplementing arrays with unused constant bytes,
reassembling the array from the pieces before use.

Another tactic to remove detections is randomization of function addresses in the final .exe file.
This can be accomplished by simply shuffling the list of object files on the linker command line:

link.exe /out:file.exe foo.obj bar.obj --> detect
link.exe /out:file.exe bar.obj foo.obj --> no detection

CHAIN DISTRIBUTION.

When bypassing AB in the case of multi-stage startup loads, almost the only way to avoid detection is to break the startup chain.
So the parent process of subsequent stages is not the previous stage, but a legitimate OS file.
For example, we want to download and run the second stage of the load from the loader.
If we do this directly, AB will detect a connection between the loader and the load.
If we add an intermediate link to the chain (e.g. start the load with an AT command), the chain will be broken.

               V. BACKCHANNEL AND COMMUNICATION

C&C servers are always hidden: either behind pads (reverse proxies) or behind a tor-domain.
The pads are cascaded, and many of them are made, so that the failure of one does not lead to the collapse of the whole network.

When implementing a communication channel with C&C, it should be remembered that the OS may have its own system proxy configured.

For initial C&C search you can use DGA (Domain Generation Algorithm)
http://www.marc-blanchard.com/BotInvaders/index.php
The point of it is to generate pseudo-random domain names.
- which are not too many (up to 10k) to be searched in a reasonable time;
- there are enough of them that they can't be squatted or otherwise banned/spoofed;
- the list of domains for one month is different from the list of domains for another month;
- it is difficult to make a regular to cut them out on dns-servers.

Usually HTTPS is used to communicate with C&C, but not always the right port or protocol is open.
If the DPI filter cuts HTTP(s)-traffic, other protocols are used:
- you can make specific DNS queries to _necessary_ (your) DNS server, and hide information in domain names
- you can make specific ICMP parcels ***
- SMTP/IMAP/POP3 mail protocols are used

In short, you can use different variants of modulation of the useful signal over the carrier,

If all channels inside the network are completely closed, the information can still be sent.

One of the options is mail (SMTP):

- Look for an SMTP server in the local machine (although they now always require authentication).
- You can also have webmail available on the local machine (e.g. OWA = Outlook Web Access) with the internet closed.

Another option is the so-called "rocket":
https://www.blackhat.com/docs/us-17/thursday/us-17-Kotler-The-Adventures-Of-Av-And-The-Leaky-Sandbox.pdf
which, as the name implies, consists of two stages.
The first stage should be Fully Undetectable.
It collects information from the system, generates a message, and flashes it to the second stage.
The second stage is a binary with the following properties:
- it annoys AB (there are signature detections)
- it knows how to contact C&C-server and send it a message flashed by carrier.

Further events develop as follows:
- corporate AB sends the second stage to the cloud for execution in the sandbox
- The second stage gets into the sandbox and is launched

               VIII. AUXILIARY SERVICE TECHNICIANS

EXCEPTIONS AND POSTMORTEM STACKING

Bug catching and debugging on an industrial scale is a fairly trivial task, but the solutions are not known to everyone and are not always trivial.
The main way to find out what went wrong is to capture the postmortem call stack and send it by telemetry to the server.
Theory and practice of exception handling in Windows: https://habr.com/ru/post/536990/.
To do that, you need to remove the postmortem stack.
To do this, you need to catch an emergency program crash (a.k.a. crash) and capture the necessary information before letting it die.
There are two main ways to do this:
* VEH - Vectored Exception Handling
* SEH - Structured Exception Handling
On Linux/*nix there are signals (SIGBUS, SIGSTOP, SIGILL, etc.), man signal

And again:
- VEH is AddVectoredExceptionHandler() and the article is https://docs.microsoft.com/ru-ru/windows/win32/debug/using-a-vectored-exception-handler
- SEH is __try ... __except and the article https://docs.microsoft.com/en-us/cpp/cpp/try-except-statement?view=vs-2019.
If SEH is used, all main threads should be wrapped in try/except.
If VEH is used, it is enough to set one common handler in the program prologue.

Among the disadvantages of SEH - Error C2712 Cannot use __try in functions that require object unwinding, and the solution is described here.
https://stackoverflow.com/questions/51701426/cannot-use-try-in-functions-that-require-object-unwinding-fix
Properties / C/C++ / Code Creation / Enable C++ Exceptions: None

The main drawback of any approach is that with process hollowed processes, no method will give line numbers and function names,
because character loading will not work. There will be only bare addresses.

In the exception handler, we must remove the stack (code below).
If we want code line numbers, we need symbols (.pdb), and the project must be built with the following options
- C/C++ / General / Debug Information Format: Program Database (/Zi)
- Composer / Debug / Create debugging information: Optimize for debugging (/DEBUG)
- Compiler / Debug / Create full program database file: Yes
and the .pdb must lie BACK INCLUDED with the dying .exe or .dll.

This won't work for combat builds, but for debugging on internal resources you can do it this way. For combat builds, the stack will just contain addresses, which is also a big deal.

The code of stack removal is quite small, so we will give it here:

#include <windows.h>
#include <Psapi.h>

// Some versions of imagehlp.dll lack the proper packing directives themselves
// so we need to do it.
#pragma pack( push, before_imagehlp, 8 )
#include <imagehlp.h>
#pragma pack( pop, before_imagehlp )

#pragma comment(lib, "psapi.lib")
#pragma comment(lib, "dbghelp.lib")


__declspec(noinline) DWORD DumpStackTrace() {
    unsigned int i;
    void * stack[100];
    unsigned short frames;
    SYMBOL_INFO * symbol;
    HANDLE process;

    debug_printf("PROGRAM CRASHED, STACK TRACE FOLLOWS:\r\n");

    process = GetCurrentProcess();

    if (!SymInitialize(process, NULL, TRUE))
        return 0;

    DWORD symOptions = SymGetOptions();
    symOptions |= SYMOPT_LOAD_LINES | SYMOPT_UNDNAME | SYMOPT_LOAD_ANYTHING | SYMOPT_CASE_INSENSITIVE;
    SymSetOptions(symOptions);

    frames = CaptureStackBackTrace(0, 100, stack, NULL);
    symbol = (SYMBOL_INFO *)calloc(sizeof(SYMBOL_INFO) + 256 * sizeof(char), 1);
    symbol->MaxNameLen = 255;
    symbol->SizeOfStruct = sizeof(SYMBOL_INFO);

    DWORD offset_from_symbol = 0;
#ifdef _WIN64
    IMAGEHLP_LINE64* line = (IMAGEHLP_LINE64*)calloc(sizeof(IMAGEHLP_LINE64), 1);
    line->SizeOfStruct = sizeof(IMAGEHLP_LINE64);
#else
    IMAGEHLP_LINE* line = (IMAGEHLP_LINE*)calloc(sizeof(IMAGEHLP_LINE), 1);
    line->SizeOfStruct = sizeof(IMAGEHLP_LINE);
#endif


    for (i = 0; i < frames; i++)
    {
        SymFromAddr(process, (DWORD64)(stack[i]), 0, symbol);

        SymGetLineFromAddr(process, (DWORD64)(stack[i]), &offset_from_symbol, line);

        debug_printf( "%i: %s (%s:%i) - 0x%0X\n", frames - i - 1, symbol->Name,
            line->FileName, line->LineNumber, symbol->Address);
        symbol->Name->Name[0] = 0;
        symbol->Address = 0;
        if(line->FileName)
            line->FileName[0] = 0;
        line->LineNumber = 0;
    }

    free(symbol);
    free(line);

    return 1;
}

DISABLING SECURITY CHECKS

Microsoft compiler puts a lot of additional security code into the final code - stack canaries, array exit checks,
nulling variables at the input to the function. All this gives extra kilobytes and is unnecessary in combat code.
- Properties / C/C++ / Code Creation / Basic runtime checks (/RTC) - by default (it is not very clear what to set here)
- Properties / C/C++ / Code Creation / Control Flow Protection - None
- Properties / C/C++ / Code Creation / Create image with update - None (only if you do not put hooks on your own functions)
- Properties / C/C++ / Language / Delete code and data not referenced - Yes /Zc:inline)
- Properties / C/C++ / Language / Include information about runtime types - No


OPTIMIZATIONS

- Omit frame pointers - by default, when entering a function, the current stack top is in the BSP register.
This way you can delimit the current function's private stack from the stack of higher-level functions.
Disabling stack frame saving frees up one more register and reduces the number of assembly instructions,
both by not writing to it and by the fact that more variables can be stored and passed through registers.

MICROSOFT RICH HEADER

http://ntcore.com/files/richsign.htm
http://bytepointer.com/articles/the_microsoft_rich_header.htm
A non-state section in the PE header, which has been inserted by Microsoft linkers since 1998 and Microsoft Visual Studio 6.0.
It contains statistics about the tulchein that built the given binary, such as - number of C object files,
number of C++ object files, number of ASM object files, linker version, resource compiler version,
number of imported functions, and so on.
This was probably done for debugging purposes (to debug the build toolchain).
However, you should realize that this header can be used for forensics as a fingerprint.
